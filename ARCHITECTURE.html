1. System Overview

Frontend

Next.js 15 App Router, React Server Components, Tailwind CSS, shadcn/ui. All pages are server-rendered by default.

RSC Server Actions Tailwind

Backend

Server Actions for mutations. API routes for auth (Better Auth), Stripe webhooks, and file uploads (R2). No middleware.ts — auth guards are per-page via requireRole().

Server Actions API Routes Better Auth

Infrastructure

Deployed on Vercel. PostgreSQL on Neon (serverless). File storage on Cloudflare R2. Stripe for payments.

11. Authorization System

Comprehensive role-based access control (RBAC) with 7 user roles and server-side enforcement

Authorization Middleware (lib/authz.ts)

getCurrentSession()

Retrieves current user session without requiring authentication. Used for optional personalization.

const session = await getCurrentSession();
// Returns: Session | null
Optional Auth Marketing Pages

requireSession()

Enforces authentication and email verification. Redirects to /login if not authenticated, /verify-email if email not verified.

const session = await requireSession();
// Throws redirect if unauthorized
Required Auth Email Check

requireRole(roles: Role | Role[])

Validates session AND checks if user's role matches. Supports single role or array of roles. Redirects to / if unauthorized.

await requireRole("super_admin");
await requireRole(["admin", "super_admin"]);
await requireRole(["participant", "admin"]);
Role-Based Primary Guard

requireStaffSession()

Enforces staff-only access (super_admin, admin, coordinator). Redirects to /staff if not authenticated, / if not staff.

const session = await requireStaffSession();
// Staff roles: super_admin, admin, coordinator
Staff Only Admin Portal

Email Verification Requirements

Protected by Email Verification

These portals require emailVerified = true (enforced by requireSession()):

participant All /participant/* routes
partner_contact All /partner/* routes

If emailVerified = false, user is redirected to /verify-email and cannot proceed until verified.

Authorization Patterns

Pattern 1: Layout-Level Authorization

Protect entire route groups by adding authorization to layout:

// app/(dashboard)/layout.tsx
export default async function DashboardLayout() {
  await requireSession(); // All nested routes protected
  return <DashboardShell>{children}</DashboardShell>;
}

Pattern 2: Page-Level Authorization

Add specific role requirements to individual pages:

// app/(dashboard)/admin/users/page.tsx
export default async function UsersPage() {
  await requireRole(["super_admin"]);
  const users = await db.query.users.findMany();
  return <UsersTable users={users} />;
}

Pattern 3: Data-Scoped Authorization

Scope database queries based on user role:

const session = await requireRole(["participant", "admin"]);
const isAdmin = session.user.role === "admin";

// Participants see only their data, admins see all
const results = isAdmin
  ? await db.query.sessions.findMany()
  : await db.query.sessions.findMany({
      where: eq(sessions.userId, session.user.id)
    });

Pattern 4: Server Action Guards

Protect data mutations with authorization checks:

"use server";
export async function deleteUser(userId: string) {
  await requireRole(["super_admin"]); // Auth guard
  await db.delete(users).where(eq(users.id, userId));
  revalidatePath("/admin/users");
}

Security Best Practices

✅ Server-Side Only

All authorization checks in Server Components and Server Actions. Never client-side.

✅ Defense in Depth

Authorization at layout, page, server action, and query levels.

✅ Fail Securely

Default to denying access. Explicit role checks required.

✅ Email Verification

Required before accessing participant and partner portals.

✅ Audit Logging

Sensitive operations logged to audit_logs table.
